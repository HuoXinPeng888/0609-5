# 系统架构约束

## 1. 数据一致性
- Redis 与数据库之间必须有补偿机制，不能只写一端
- 任何写 Redis + 写 DB 的操作，DB 失败时必须回滚 Redis

## 2. 分布式锁
- 分布式锁必须使用 `SET key value NX EX seconds` 单条命令（原子操作）
- 禁止 SETNX + EXPIRE 两步操作（非原子，有竞态风险）

## 3. 幂等性
- 所有涉及资金的操作（支付、退款）必须有幂等性控制
- 使用唯一的业务 ID（如 orderId + 操作类型）作为幂等 key
- 重复请求应返回已有结果而不是重复执行

## 4. 消息可靠性
- 消息消费必须"先处理，后 ACK"
- 处理失败的消息必须进入死信队列（dead letter queue）
- 消费者重启后必须能恢复未 ACK 的消息

## 5. 订单状态机
- 所有非终态必须有超时退出机制
- PAYING 状态必须有超时自动取消（释放库存 + 关闭支付）
- 超时时间通过配置管理，不能硬编码

## 6. 金额精度
- 金额计算使用 BigDecimal，RoundingMode 统一为 HALF_UP
- 禁止使用 double/float 做金额运算
